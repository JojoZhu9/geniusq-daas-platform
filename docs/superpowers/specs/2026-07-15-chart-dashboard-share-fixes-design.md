# 图表、仪表盘移动与分享修复设计

## 背景

人工验收确认了三个缺陷：2.4-a 的“折线图”仍是变色后的横向条形；2.6 虽保存卡片 `x` 坐标，但页面没有使用该坐标；复制出的 `/share/{shareId}` 链接没有对应前端页面。本设计只修复这三条验收路径，不改变后端分析、知识库或样例数据。

## 目标与验收标准

1. 折线、柱状和表格使用三种真实渲染形态。折线与柱状由已有 ECharts 5.6 的 SVG renderer 输出，切换后 DOM 与视觉形态均发生变化。
2. 仪表盘卡片使用后端返回的 `x/y/w/h` 设置 12 列 CSS Grid。点击“移动卡片”后卡片在左、右位置间切换，刷新后位置不回退。
3. `/share/:shareId` 是无需编辑能力的只读分享页，调用已有 `GET /api/dashboards/share/{shareId}`，展示看板名称、分享标识和卡片；不存在的分享标识显示明确错误。

## 方案比较与选择

### 图表

- 选择：使用项目已安装的 ECharts core，按需注册 LineChart、BarChart、Grid、Tooltip、Legend、Aria 和 SVGRenderer。优点是真实图表语义、后续可扩展、SVG 可测试；成本是增加前端 bundle。
- 未选：继续用 CSS 拼装。代码少，但需要自行维护坐标轴、折线、提示和多序列，容易再次出现“形态只换颜色”的问题。
- 未选：引入新的图表依赖。没有必要，项目已经包含 ECharts。

### 卡片移动

- 选择：保留当前“移动卡片”按钮，把目标 `x` 计算为 `0` 或 `12 - w`，并把 `x/y/w/h` 映射到 `gridColumnStart/gridRowStart/gridColumnEnd/gridRowEnd`。它与现有 API 完全兼容，且能稳定自动验收。
- 未选：本轮引入拖拽库。拖拽体验更强，但会扩大范围并引入碰撞、键盘可访问性与新依赖问题。

### 分享

- 选择：增加独立只读 React 页面和 `/share/:shareId` 路由，继续使用后端现有分享 API。
- 未选：把分享页放在带侧边栏的平台壳中。公开查看者不需要编辑导航，独立页面更符合分享语义。

## 组件与数据流

- `AnalysisChart`：把后端 `ChartSpec + Dataset[]` 转为 ECharts option；房价月度数据按 `district` 生成多条序列，多源数据按共同 `district` 合并字段；表格继续展示原始数据。
- `DashboardWorkspace`：PATCH 布局后使用响应中的坐标重绘；卡片 DOM 暴露可访问名称与 `data-grid-x/y`，便于现场和自动化核查。
- `SharedDashboardPage`：从 URL 读取 `shareId`，请求分享 API并渲染只读卡片；加载、空看板、404 都有页面状态。
- 后端不新增接口，也不改变数据库结构。

## 错误处理

- ECharts 实例在组件更新和卸载时释放；窗口或容器尺寸变化时调用 `resize`。
- 分享请求失败时显示“分享链接无效或已失效”，不跳回智能问数页。
- 卡片移动继续使用统一 API 错误边界；位置右移时按卡片宽度限制在 12 列以内。

## 测试策略

1. Vitest：先复现折线/柱状没有真实 SVG 差异，再验证切换后 ECharts SVG 和图表类型变化。
2. Vitest：先复现移动后 `x` 未映射到样式，再验证右移和返回左侧。
3. Vitest：先复现 `/share/:shareId` 路由缺失，再验证分享页调用正确 API并且无编辑按钮。
4. Playwright：比较折线与柱状 SVG、比较移动前后卡片 `boundingBox.x`、刷新验证持久化、从剪贴板打开分享页。
5. 完整回归：前端单测、TypeScript/Vite build、Playwright、后端 Pytest和真实启动健康检查。

## 范围边界

- 本轮不加入自由拖拽、卡片碰撞自动排版、权限鉴权、分享密码或过期时间。
- 分享页展示已持久化的图表规格与只读预览；当前卡片模型没有保存原始数据快照，因此不在分享页重新执行分析 SQL。
